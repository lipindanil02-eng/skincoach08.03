## 1. Тройное дублирование пайплайна (архитектурное)

**Проблема:** В проекте существует **три независимые реализации** 8-слойного LLM-пайплайна:
1. `bot.py` (строки 127–245) — встроенный в Telegram-бота
2. `core/pipeline.py` (полный файл, 390 строк) — выделенный модуль
3. `web/backend/pipeline.py` (полный файл, 207 строк) — веб-версия

Каждая копия содержит свою версию `pipeline_photo`, `pipeline_final`, `call_raw`, `rp`, `xj`/`cm`, констант WEEKS/W_EMOJI/FOCUSES.

**Риски:** Багфиксы нужно делать в 3 местах. Расхождение логики неизбежно. Увеличение размера кода, сложность поддержки.

**Предложение:** Удалить пайплайн из `bot.py` и `web/backend/pipeline.py`, оставив единственную реализацию в `core/pipeline.py`. `bot.py` должен импортировать `pipeline_photo`/`pipeline_final` оттуда. `web/backend/analyze.py` уже импортирует из `core.pipeline` — его не трогать, а `web/backend/pipeline.py` удалить.

**Оценка:** ~150 строк изменений, регрессионное тестирование обязательно.

## 2. Отсутствие unit-тестов (качество)

**Проблема:** Директория `tests/` пуста. В проекте есть:
- `test_pipeline.py` — интеграционный тест, требует фото
- `test_pipeline_final.py` — интеграционный, требует API-ключ
- `web/backend/test_backend.py` — TestClient, требует БД

Нет юнит-тестов для:
- `xj()` (парсинг JSON из ответов LLM)
- `cm()` (очистка markdown)
- `rp()` (чтение промптов)
- `format_fallback()` (формирование ответа)
- `predict_image()` (ML-инференс)
- Моделей данных (User, Analysis, ProgramProgress)
- Роутеров (auth, profile, program)

**Предложение:** Написать базовый набор pytest-тестов для core-модулей (без внешних зависимостей) — минимум 10–15 тестов.

**Оценка:** ~300 строк нового кода.

## 3. Дублирование загрузки ML-модели (inference + web/ml_service)

**Проблема:** Одна и та же EfficientNet-B3 модель загружается двумя разными способами:
- `inference.py` — прямая загрузка через `torch.load` с HTTP-стримом из HuggingFace
- `web/ml_service/main.py` — через `huggingface_hub.hf_hub_download`

Разные пути к model path, разные обработчики ошибок, разный код создания архитектуры.

**Предложение:** Вынести загрузку модели в общий модуль (например, `core/model_loader.py`) и переиспользовать его в `inference.py` и `web/ml_service/main.py`.

**Оценка:** ~80 строк нового кода + рефакторинг двух файлов.

## 4. httpx-клиент создаётся на каждый вызов (производительность)

**Проблема:** В `call_raw` (и в `bot.py`, и в `core/pipeline.py`, и в `web/backend/pipeline.py`) новый `httpx.AsyncClient()` создаётся на каждую итерацию цикла fallback-моделей (строки 101, 152, 42).

**Последствия:** На каждый запрос к OpenRouter создаётся новое TCP-соединение, новый SSL-handshake. При типичном 3–5 fallback-моделях это 3–5 соединений на один вызов пайплайна × 8 шагов = 24–40 соединений на один анализ фото.

**Предложение:** Создавать один `httpx.AsyncClient` на время выполнения `call_raw` и переиспользовать его для всех попыток в цикле (вынести `async with` за пределы цикла).

**Оценка:** ~30 строк изменений в 3-х файлах.

## 5. Разделение конфигурации (SPOF)

**Проблема:** OpenRouter API-key, модели, fallback-модели, температуру и таймаут нужно менять в 4 местах:
- `bot.py` (строки 18–26)
- `core/pipeline.py` (строки 16–26)
- `web/backend/analyze.py` (строка 32)
- `web/backend/pipeline.py` (строки 17–26)

**Предложение:** Вынести в единый `core/config.py`, читающий из `.env`. Остальные модули импортируют оттуда.

**Оценка:** ~50 строк нового кода + изменения в 4 файлах.

## 6. Два генератора карточек (неоднозначность)

**Проблема:** В проекте два разных способа генерации карточек результатов:
- `gen_card.py` — через PIL (ImageDraw), сохраняет JPEG в `static/share_card.jpg`. Один выходной файл на всех пользователей.
- `share_card.py` — через Chrome headless + HTML-шаблон, сохраняет PNG в `static/cards/card_{user_id}.png`. Индивидуальные карточки, но требует Chrome.

`share_card.py` не имеет обработки ошибок на случай отсутствия Chrome.

**Предложение:** Выбрать один способ, удалить другой. Рекомендуется PIL-подход (gen_card.py) без внешних зависимостей от Chrome. Добавить user_id в имя файла.

**Оценка:** ~40 строк изменений.
